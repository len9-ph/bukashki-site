# Букашки сайт — C4 workshop notes

## Зачем этот файл

Этот файл фиксирует промежуточные решения по проектированию pet-проекта "Букашки сайт" в терминах C4.
Цель — не написать идеальную архитектуру сразу, а постепенно прояснить:
- контекст системы
- границы продукта
- ключевые пользовательские сценарии
- внешние зависимости
- направление для следующих C4-уровней

---

# 1. Общая идея проекта

## Рабочее название
**Букашки сайт**

## Идея проекта
Pet-проект про цифровую коллекцию насекомых.

Изначальная идея вдохновлена реальными биологами, которые собирают насекомых на сборах.  
Проект задуман как учебный полигон, где можно потрогать:
- качественный backend
- проектирование домена
- права доступа
- процессы обмена
- интеграции
- возможно, фронтенд на React

## Технологический ориентир
- Backend: Java + Spring Boot
- Database: PostgreSQL
- Frontend: пока не определен, возможен React
- Основной фокус: **качественный backend**

## Целевой уровень проекта
Не примитивный CRUD и не production-уровень, а **что-то среднее**:
достаточно серьезный pet-проект с хорошей архитектурной базой.

# 3. C4 Level 1 — System Context

## Вопросы, которые были поставлены

Для уровня System Context нужно ответить прежде всего на вопросы:

1. Кто главный пользователь или пользователи системы?
2. Какую ценность система им дает?
3. Есть ли внешние системы кроме самой системы?
4. Где проходят границы самой системы?

Дополнительно были заданы уточняющие вопросы:
- пользователь хранит виды или конкретные экземпляры?
- нужны ли фото?
- обмен — это просто действие или процесс?
- кто может видеть коллекции других пользователей?
- нужна ли дружба/социальные связи?

---

## Черновые ответы пользователя

### Кто пользователи
Пользователи — это люди-биологи, которые собирают насекомых.

### Какая ценность
Иметь цифровую коллекцию своих насекомых и смотреть, какие есть у других.

### Про внешние системы
Сначала не было понимания, какие внешние системы нужны.  
Позже были названы возможные кандидаты:
- авторизация
- внешний справочник видов
- карта
- хранение картинок

### По модели данных на уровне идеи
- хранить нужно **экземпляры**, а не просто виды
- фото нужны
- обмен — это **процесс**
- хочется добавить друзей и управление тем, кому видна коллекция

---

## Что обсуждали по ходу

### 1. Виды vs экземпляры
Было решено, что ядро проекта должно строиться вокруг **экземпляров насекомых**, а не вокруг абстрактного списка видов.

Причина:
- это лучше соответствует реальной биологической коллекции
- это делает осмысленными фото, метаданные и обмен
- это делает домен живее и интереснее

### 2. Фото
Фото признаны важной частью модели.

### 3. Обмен
Обмен не должен быть просто кнопкой "отдать", а должен быть **процессом**.

### 4. Видимость коллекции
Появилась идея приватности и социальной модели:
- добавление в друзья
- управление тем, кому видна коллекция

### 5. Внешние системы
Разобрали, что внешняя система в C4 — это не пользователь и не часть приложения, а отдельная система, с которой приложение взаимодействует.

---

## Выводы по внешним системам

### Авторизация
Уточнение:
- если логин/пароль внутри приложения, это **не внешняя система**
- если вход через Google/GitHub/OAuth provider, это **внешняя система**

На данном этапе разумнее считать, что базовая авторизация будет частью самой системы.

### Внешний справочник видов
Возможен как будущая интеграция, но не обязателен в первой версии.  
Более практично на старте иметь собственный справочник видов внутри системы.

### Карта
Хорошая идея для развития проекта.  
При этом:
- место находки и координаты могут быть частью домена
- картографический сервис становится внешней системой только если реально есть интеграция

### Картинки
Идея хранения фото признана важной.  
Архитектурно разумно мыслить через отдельное файловое/объектное хранилище, даже если первая реализация будет простой.

---

## К чему пришли на уровне System Context

### Основной пользователь
**Коллекционер насекомых / биолог**

Он:
- ведет цифровую коллекцию экземпляров насекомых
- просматривает коллекции других пользователей
- настраивает видимость своей коллекции
- добавляет пользователей в друзья
- инициирует и участвует в обменах

### Система
**Букашки сайт**

Это веб-система, которая позволяет:
- хранить экземпляры насекомых
- прикладывать фото и метаданные
- управлять своей коллекцией
- просматривать чужие коллекции с учетом правил доступа
- организовывать обмены между пользователями

### Внешние системы-кандидаты
- **Email Service** — уведомления, регистрация, восстановление доступа
- **Image Storage** — хранение фотографий
- **External Species Catalog** *(опционально)* — справочник видов
- **Map / Geocoding Service** *(опционально)* — карта и работа с местами находок

# 4. Зафиксированные продуктовые решения после Level 1

На текущий момент как базовая версия проекта зафиксированы такие идеи:

## Доменное ядро
- пользователь
- коллекция
- экземпляр насекомого
- вид насекомого
- фото экземпляра
- место находки / геоданные (возможно)
- обмен

## Социальные и access-control элементы
- друзья
- видимость коллекции
- просмотр чужих коллекций по правилам доступа

## Предварительная модель видимости
Как рабочая идея:
- `PRIVATE`
- `FRIENDS_ONLY`
- `PUBLIC`

## Предварительная модель обмена
Обмен рассматривается как процесс со статусами.  
Черновой пример статусов:
- `CREATED`
- `ACCEPTED`
- `DECLINED`
- `CANCELED`
- `COMPLETED`

# 5. Что сознательно не включаем пока

Чтобы не перегрузить проект слишком рано, пока не берем в первую волну:
- сложную модерацию
- большое количество ролей
- чат между пользователями
- real-time уведомления
- аукционы / marketplace
- слишком сложную биологическую таксономию
- продвинутые интеграции без необходимости

---

# 6. Что стало понятно после Level 1

После обсуждения System Context стало понятно следующее:

1. Это не просто CRUD по "насекомым", а система с доменной моделью.
2. Центральная сущность на уровне идеи — не "вид", а **экземпляр**.
3. Фото и обмены делают систему намного содержательнее.
4. Приватность коллекции и дружба добавляют реальную бизнес-логику.
5. Внешние системы можно добавлять постепенно, не пытаясь интегрировать все сразу.
6. Теперь есть достаточная база для перехода к **C4 Level 2 — Container Diagram**.

Следующий уровень должен ответить на вопросы:

1. Из каких крупных технических частей состоит система?

- Web Frontend
- Backend API
- PostgreSQL
- Image Storage
- Email integration

# 8. Текущее краткое резюме

"Букашки сайт" — это pet-проект средней сложности про цифровую коллекцию экземпляров насекомых.  
Основной пользователь — биолог/коллекционер.  
Система должна позволять хранить экземпляры с фото и метаданными, смотреть чужие коллекции, управлять приватностью, добавлять друзей и проводить обмены.  
На уровне C4 Level 1 уже определены:
- основной пользователь
- назначение системы
- границы системы
- потенциальные внешние интеграции
- ключевые продуктовые решения

Следующий шаг — C4 Level 2: выделение контейнеров системы.
