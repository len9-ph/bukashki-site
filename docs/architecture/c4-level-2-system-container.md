# C4 Level 2 — Container Diagram
## Проект: Букашки сайт

## 1. Назначение документа

Этот документ фиксирует архитектуру системы **«Букашки сайт»** на уровне **C4 Level 2 (Container Diagram)**.

Цель этого уровня:
- выделить основные технические части системы
- определить границы между ними
- зафиксировать связи между контейнерами
- кратко описать назначение каждого контейнера

На этом уровне **не рассматриваются**:
- внутренние модули backend-приложения
- классы
- пакеты
- детали схемы базы данных

---

## 2. Общая архитектурная идея

Первая версия системы рассматривается как **веб-система с отдельным frontend, backend, реляционной базой данных и выделенным файловым хранилищем**.

Система строится как единое приложение с явным разделением ответственности:
- **Web App** отвечает за взаимодействие с пользователем
- **Backend API** содержит основную прикладную и бизнес-логику
- **PostgreSQL Database** хранит структурированные данные системы
- **File Storage** хранит файлы, используемые системой

---

## 3. Контейнеры системы

### 3.1 Web App
**Технология:** React

**Назначение:**  
Пользовательский интерфейс системы.

**Ответственность:**
- отображение данных, полученных от backend
- показ страниц входа и регистрации
- отображение профиля пользователя
- отображение коллекций и экземпляров
- отображение фотографий
- работа со страницами друзей и обменов
- отправка действий пользователя на backend

**Важно:**  
Web App не содержит основной бизнес-логики и не принимает архитектурных решений о доступе, правах и допустимости операций.

---

### 3.2 Backend API
**Технология:** Java, Spring Boot, MyBatis

**Назначение:**  
Центральный контейнер системы, реализующий прикладную и бизнес-логику.

**Ответственность:**
- работа с пользователями и локальной авторизацией
- обработка запросов от Web App
- работа со справочной информацией по видам
- работа с конкретными экземплярами насекомых
- управление доступом к коллекциям
- работа с файлами и их метаданными
- социальные взаимодействия: друзья и обмены
- взаимодействие с PostgreSQL Database
- взаимодействие с File Storage

**Важно:**  
Backend является единственной частью системы, которая принимает решения по правилам доступа и жизненному циклу данных и файлов.

---

### 3.3 PostgreSQL Database
**Технология:** PostgreSQL

**Назначение:**  
Хранение структурированных данных системы.

**Хранимые данные:**
- пользователи
- учетные данные
- пользовательские настройки
- справочники видов
- экземпляры насекомых
- файлы и связи файлов с доменными сущностями
- данные о дружбе между пользователями
- данные об обменах

---

### 3.4 File Storage
**Технология:** локальное файловое хранилище с возможностью последующей замены реализации

**Назначение:**  
Предоставляет backend API для хранения и получения файлов, независимо от конкретной реализации хранилища.

**Ответственность:**
- сохранение файлов
- получение файлов
- удаление файлов

**Первая версия:**
- файлы хранятся в локальной папке, задаваемой при запуске приложения

**Перспектива развития:**
- реализация хранилища должна быть заменяемой без существенного изменения остальной системы
- в будущем локальное хранилище может быть заменено на облачное или иное внешнее файловое хранилище

**Типы файлов первой версии:**
- фотографии экземпляров насекомых
- аватары пользователей

---

## 4. Связи между контейнерами

### User → Web App
Пользователь взаимодействует с системой через веб-интерфейс.

### Web App → Backend API
Web App отправляет запросы на backend и получает от него данные для отображения.

### Backend API → PostgreSQL Database
Backend читает и изменяет структурированные данные системы.

### Backend API → File Storage
Backend сохраняет, получает и удаляет файлы через абстракцию хранилища.

---

## 5. Границы системы

На текущем этапе все перечисленные контейнеры считаются **частью самой системы «Букашки сайт»**:
- Web App
- Backend API
- PostgreSQL Database
- File Storage

Внешние интеграции, такие как email-сервисы, карты или внешние справочники видов, в первую версию Container Diagram не включаются.

---

## 6. Архитектурные решения, зафиксированные на этом уровне

### 6.1 Web App не принимает бизнес-решения
Frontend отвечает только за отображение данных и отправку действий пользователя.  
Все проверки доступа, разрешений и правил происходят на backend.

### 6.2 Backend является центром прикладной логики
Именно backend отвечает за:
- авторизацию
- доступ к коллекциям
- работу с экземплярами
- дружбу
- обмены
- работу с файлами

### 6.3 File Storage выделен как отдельный контейнер
Хотя первая реализация будет локальной, файловое хранилище выделено отдельно уже сейчас, чтобы:
- явно показать техническую границу
- отделить файловую инфраструктуру от backend-логики
- упростить будущую замену локального хранения на облачное

### 6.4 Backend работает со Storage только через абстракцию
Логика приложения не должна зависеть от конкретной файловой системы.  
Текущая реализация локальная, но контракт хранилища должен позволять заменить реализацию позже.

### 6.5 Хранилище не знает доменную модель
File Storage не должен знать о пользователях, экземплярах, друзьях или обменах.  
Он работает только с файлами.

---

## 7. Итог

На уровне **C4 Level 2** система **«Букашки сайт»** зафиксирована как веб-система, состоящая из:
- **Web App**
- **Backend API**
- **PostgreSQL Database**
- **File Storage**

Эта модель отражает текущую архитектуру первой версии проекта и подготавливает основу для следующего шага — детализации внутреннего устройства **Backend API** на следующем уровне C4.
